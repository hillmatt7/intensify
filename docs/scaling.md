# Scaling study

How does fit time scale with dataset size? This page answers that
directly for multivariate exp-Hawkes (M=5) against tick in decay-given
mode — the apples-to-apples setting.

All datasets were generated by the multivariate branching (Galton-Watson)
simulator, seeded with `42`, fit with β fixed to ground truth
(`tick` requires this; `intensify` opts in via `fit_decay=False`).
Runs are median of 3 timings after a warm-up fit. Reproduce:

```bash
# Build the Rust extension once
maturin develop --release

# Generate scale datasets (one-time, ~1 min via branching sim)
.venv/bin/python benchmarks/reference_dataset.py \
    mv_exp_5d_scale_small mv_exp_5d_scale_medium \
    mv_exp_5d_scale_large mv_exp_5d_scale_xl mv_exp_5d_scale_xxl

# Run each library
.venv/bin/python benchmarks/run_scaling.py --lib intensify
micromamba run -n tickbench38 python benchmarks/run_scaling.py --lib tick
```

## Results — `intensify 0.3.2` (Rust core) vs `tick 0.7.0.1`

| N total | intensify time | intensify RMSE | tick time | tick RMSE | speedup |
|---:|---:|---:|---:|---:|---:|
| 501 | **0.5 ms** | 0.072 | 1.1 ms | 0.082 | **2.0× faster** |
| 2,249 | **0.8 ms** | 0.037 | 2.2 ms | 0.037 | **2.8× faster** |
| 9,271 | **2.0 ms** | 0.016 | 5.7 ms | 0.016 | **2.8× faster** |
| 27,519 | **5.5 ms** | 0.012 | 15.0 ms | 0.013 | **2.8× faster** |
| 91,249 | **17.5 ms** | 0.006 | 47.8 ms | 0.006 | **2.7× faster** |

For reference, the same numbers under intensify 0.2.0 (JAX-on-CPU):

| N total | 0.2.0 time | 0.3.2 time | port speedup |
|---:|---:|---:|---:|
| 501 | 8 ms | 0.5 ms | 16× |
| 9,271 | 38 ms | 2.0 ms | 19× |
| 91,249 | 549 ms | 17.5 ms | 31× |

## What the curve says

**Both libraries scale linearly in N.** At the 91 k-event point
intensify's per-event cost is ~0.19 µs; tick's is ~0.52 µs. The
recursive likelihood (per-target weight precomputation pattern from
tick's own `ModelHawkesExpKernLogLikSingle`, ported to safe Rust)
keeps the compute asymptotics correct and the constant factor below
tick's.

**The speedup ratio is stable at ~2.0–2.8× across all benchmarked N.**
It does not grow with N — that's expected for two well-tuned
implementations of the same O(N·M) algorithm. The remaining gap
to a hypothetical "tick + per-target weights + L-BFGS in Rust" is
the scipy.optimize.minimize wrapper overhead (each L-BFGS-B
iteration round-trips into Python); closing it would require
porting the optimizer too, which is out of scope until a user reports
needing it.

**Parameter-recovery RMSE matches the 0.2.0 baseline** (which already
matched tick to within noise from N≈9k onward). The optimizer is
unchanged across the 0.2.0 → 0.3.0 port; only the inner-loop value+
gradient computation moved from XLA-on-CPU to Rust. Asymptotic
consistency holds.

## Practical implication

For realistic lab datasets (thousands to millions of events),
intensify finishes faster than tick at every benchmarked size while
preserving (and at small N improving on) parameter-recovery
accuracy. **Choose on capabilities first** — kernel variety, joint-
decay fits, marked / nonlinear support, Python 3.10+ wheels — but
the speed argument no longer cuts the other way.

Out of scope for now:

- Rust-native L-BFGS-B (would close the small-N residual gap).
- GPU port (the recursive likelihood is sequential; gains would be
  from batch fitting many series at once, not single-series speed).
